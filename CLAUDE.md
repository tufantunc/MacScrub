# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

MacScrub is a macOS (14+) menu-bar utility that temporarily blocks all keyboard and trackpad input so the user can wipe down their Mac. It's a menu-bar accessory (`LSUIElement`, no Dock icon) but still shows a main window on launch. Pure SwiftUI + AppKit, Swift 5.9, **zero third-party dependencies**.

## Commands

The Xcode project is **generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml`** — `project.yml` is the source of truth, not `MacScrub.xcodeproj`.

```bash
# Regenerate the Xcode project after adding/removing/renaming files or editing project.yml.
# Sources are directory-globbed (sources: - MacScrub / - MacScrubTests), so NEW files are
# picked up automatically by this — do NOT hand-edit MacScrub.xcodeproj/project.pbxproj.
xcodegen generate

# Build
xcodebuild build -project MacScrub.xcodeproj -scheme MacScrub -destination 'platform=macOS'

# Run the full test suite
xcodebuild test -project MacScrub.xcodeproj -scheme MacScrub -destination 'platform=macOS'

# Run a single test suite or one test (Swift Testing — use the STRUCT name, not the @Suite display name)
xcodebuild test -project MacScrub.xcodeproj -scheme MacScrub -destination 'platform=macOS' \
  -only-testing:MacScrubTests/SettingsStoreTests
xcodebuild test -project MacScrub.xcodeproj -scheme MacScrub -destination 'platform=macOS' \
  -only-testing:MacScrubTests/SettingsStoreTests/testPersistenceAcrossInstances

# Build a runnable .app and launch it
rm -rf build
xcodebuild build -project MacScrub.xcodeproj -scheme MacScrub -configuration Debug -destination 'platform=macOS' -derivedDataPath build
open build/Build/Products/Debug/MacScrub.app
```

### Reading test results

Tests use **Swift Testing** (`import Testing`, `@Suite`/`@Test`/`#expect`), not XCTest. The real result is the line `Test run with N tests in M suites passed`. The legacy `Executed 0 tests` line is XCTest noise — **ignore it**; it does not mean tests didn't run.

### Commit hygiene

Building often makes Xcode's String Catalog tool reformat `MacScrub/Localization/Localizable.xcstrings` in the working tree (cosmetic key-reordering). Unless you intentionally changed strings, revert it before staging:
```bash
git checkout -- MacScrub/Localization/Localizable.xcstrings
```
Also never stage `…/xcuserdata/…/UserInterfaceState.xcuserstate`.

## Architecture

Four layers under `MacScrub/`. The spine is a set of `@Observable` model objects whose `Date`/state changes drive SwiftUI (including `TimelineView` for the animated ring/countdown), plus low-level Core services behind protocols so they can be mocked in tests.

- **`App/MacScrubApp.swift`** — `@main`. In `init()` it constructs ONE shared instance each of `SettingsStore`, `EventBlocker`, `LidMonitor`, `CleaningModeManager`, and `HubNavigation`, then injects the same instances into all scenes. Three scenes: a `Window(id: "main")` hosting `MainWindowView`, a `MenuBarExtra` (`.menuBarExtraStyle(.menu)`) hosting `MenuBarView`, and the full-screen blocking overlay which is an AppKit `NSWindow` managed by `OverlayWindowController` (also in this file), shown/hidden by the manager.

- **`State/`** — `@Observable` models. `SettingsStore` persists settings via stored properties with `didSet` writing to `UserDefaults` (NOT computed properties — see gotcha below). `CleaningModeManager` orchestrates everything: `activate()`/`deactivate()`, wiring callbacks, and the idle-reset auto-terminate timer (`idleExitDeadline` + `noteActivity()`). `HubNavigation` (`view: .main | .preferences`) switches what the single window shows. `ModifierKeyFlags` (OptionSet), `AppLanguage` (enum driving the language picker).

- **`Core/`** — `EventBlocker` installs a `CGEvent` tap that swallows keyboard/trackpad events and reports activity via `onFlagsChanged` / `onKeyActivity`. `ModifierKeyDetector` (`@Observable`) tracks `pressedKeys` and `holdStartDate` for the hold-to-exit gesture. `LidMonitor` uses IOKit power notifications. `EventBlocker` and `LidMonitor` sit behind `EventBlockerProtocol` / `LidMonitorProtocol` so tests inject mocks (`MockEventBlocker`, `MockLidMonitor` live in `CleaningModeManagerTests.swift`).

- **`Views/`** — `MainWindowView` (idle hub + in-window Preferences, switched by `HubNavigation`), `MenuBarView` (native menu), `CleaningOverlayView` (full-screen active overlay), `ModifierKeySquare` (keycap), `PermissionGuideView` (Accessibility-permission panel). `Theme.swift` holds the `MSColor` teal palette **and** the pure, unit-tested formatting helpers (`holdRemainingText`, `autoExitProgress`, `autoExitRemainingText`) that the overlay ring renders.

### Runtime data flow during cleaning

```
EventBlocker tap ─┬─ onFlagsChanged → ModifierKeyDetector.updateFlags
                  │                     → pressedKeys / holdStartDate (@Observable)
                  │                     → ModifierKeyDetector.onAllKeysHeld → manager.deactivate()
                  └─ onKeyActivity   → CleaningModeManager.noteActivity()
                                        → idleExitDeadline (@Observable) bumped forward

CleaningOverlayView ring (TimelineView):
  holding exit keys  → fills over holdDuration, shows remaining (holdRemainingText)
  idle (not holding) → fills toward idleExitDeadline, shows M:SS (autoExit* helpers)
```
Auto-terminate is **idle-reset**: any keyboard activity pushes `idleExitDeadline` forward, so it only fires after the user truly stops. Mouse activity does not count.

## Non-obvious behaviors & gotchas

These cost real debugging time; preserve them:

- **Settings must use stored-property + `didSet`, not computed `UserDefaults` accessors.** `@Observable` does not track computed properties, so the earlier computed-property `SettingsStore` silently failed to persist/update. `init` loads from `UserDefaults`; `didSet` writes back (and `didSet` correctly does not fire during `init`).

- **Code signing must be a stable identity** (`DEVELOPMENT_TEAM`/`CODE_SIGN_IDENTITY: "Apple Development"` in `project.yml`), NOT ad-hoc `"-"`. macOS TCC keys the Accessibility grant to the signature's designated requirement; ad-hoc signing pins it to the per-build cdhash, so every rebuild invalidated the grant and re-prompted. Don't revert to ad-hoc.

- **`LSUIElement: true`** (set in `project.yml`'s `info.properties`, so it survives regeneration) makes the app a menu-bar accessory. The window is still shown on launch via an explicit `NSApp.activate(...)`.

- **The overlay `NSWindow` sets `isReleasedWhenClosed = false`.** Without it, closing the overlay double-frees (ARC + AppKit) and crashes on deactivate.

- **`MainWindowView` and `CleaningOverlayView` pin `.environment(\.colorScheme, .light)` and paint a light material background** (`MSColor.*` text is fixed dark-on-light). Without this they're unreadable in macOS Dark mode. The main window goes further: a behind-window `NSVisualEffectView` on a non-opaque window gives true translucency (a plain `.regularMaterial` over the opaque window reads as flat gray).

- **Localization** lives in the String Catalog `MacScrub/Localization/Localizable.xcstrings` (en / tr / zh-Hans). The in-app Preferences language picker is driven by the `AppLanguage` enum, separate from the catalog; selecting a language writes `AppleLanguages` and prompts a restart. See `docs/ADDING_A_TRANSLATION.md` for the full procedure.
